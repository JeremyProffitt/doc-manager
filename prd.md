# Doc-Manager: Product Requirements Document

## 1. Overview

Doc-Manager is an online document management tool that allows users to upload blank forms (PDFs/images), leverage AI to automatically detect optimal placement for standard form fields, and then auto-populate those forms with customer data. The system eliminates repetitive manual form-filling by mapping reusable customer information to detected fields across any uploaded form.

## 2. Problem Statement

Businesses frequently fill out the same customer information (name, address, phone, etc.) across dozens of different forms. Each form has a unique layout, requiring manual data entry every time. This is error-prone, time-consuming, and scales poorly as the number of forms and customers grows.

## 3. Solution

A web application where users:

1. **Log in** with email and password
2. **Upload** blank forms (PDF or image files)
3. **AI analyzes** each form using Amazon Bedrock (Claude Sonnet) to identify where standard fields should be placed
4. **Review and edit** the AI-suggested field placements on a dedicated form editor page, with full version history
5. **Select** a form and a customer, and the system **auto-populates** the form with that customer's data
6. **Download** the completed, filled-in form

## 4. Target Users

- Small-to-medium businesses that handle high volumes of repetitive paperwork
- Administrative staff filling out vendor forms, government forms, insurance documents, etc.
- Any user who repeatedly enters the same information across multiple form templates

## 5. Core Features

### 5.1 Authentication

- Simple email/password authentication backed by a DynamoDB `Users` table
- Login page at `/login`; unauthenticated users are redirected there
- Passwords stored as bcrypt hashes — never stored in plaintext
- Session managed via secure HTTP-only cookie with a signed JWT or session token
- No self-registration in Phase 1; users are seeded directly into DynamoDB
- Logout endpoint clears the session

| Password | `Docs4President!` (bcrypt hash stored in DynamoDB) |

> **Note:** The seed user is inserted via a setup script or SAM custom resource during initial deployment. The plaintext password is never stored — only the bcrypt hash.

### 5.2 Form Upload

- Users can upload PDF or image files (PNG, JPG, TIFF) of blank forms
- Uploads go **directly from the browser to S3** via a pre-signed PUT URL generated by the backend
- Flow:
  1. User selects a file in the browser
  2. Frontend requests a pre-signed upload URL from the backend (`POST /api/forms/upload-url`)
  3. Backend generates a pre-signed S3 PUT URL (5-minute expiry) and creates a form metadata record in DynamoDB with status `uploading`
  4. Browser uploads the file directly to S3 using the pre-signed URL
  5. Browser notifies the backend that upload is complete (`POST /api/forms/:id/upload-complete`)
  6. Backend updates the form status to `uploaded`
- This approach bypasses the Lambda 6MB payload limit entirely
- Uploaded forms are stored and associated with the authenticated user
- A form library page lists all uploaded forms with thumbnail previews
- Supported file size: up to 10MB per form

### 5.3 AI-Powered Field Detection

- Upon upload completion, the form can be sent to **Amazon Bedrock (Claude Sonnet)** for analysis
- The AI examines the form layout and identifies the best placement (x, y coordinates and dimensions) for each configured standard field
- The AI returns a confidence score for each field placement
- Detection runs asynchronously; the user is notified when analysis is complete
- If the AI cannot confidently place a field, it is flagged for manual review
- This is a POC — AI accuracy may vary on complex forms; the manual editor is the primary safety net

### 5.4 Standard Form Fields (Configurable)

Default fields provided out of the box:

| Field Name    | Type   | Description                  |
|---------------|--------|------------------------------|
| Name          | Text   | Full name of the customer    |
| Business      | Text   | Business or company name     |
| Address       | Text   | Street address               |
| City          | Text   | City                         |
| State         | Text   | State or province            |
| Zip           | Text   | ZIP or postal code           |
| Phone Number  | Text   | Primary phone number         |

- Users can add, remove, or rename standard fields from a settings page
- Custom fields are available across all forms once created
- Field configuration is stored in DynamoDB (`Settings` table)

### 5.5 Form Editor Page

- Dedicated page for reviewing and editing field placements on a form
- Displays the uploaded form as a background image/canvas
- Overlays draggable, resizable field markers on the form
- Each field marker is labeled with the field name and color-coded
- Users can:
  - **Drag** fields to adjust position
  - **Resize** fields to match the form's input areas
  - **Add** fields that the AI missed
  - **Remove** fields that are incorrectly placed
  - **Override font settings** per field (font family, size) — defaults to form-level settings
  - **Save** the final field layout — each save creates a new version
  - **View version history** and revert to any previous version
- Changes are persisted in DynamoDB and used for all future auto-population of that form

### 5.6 Font & Text Configuration

Font and text rendering settings are configurable at **two levels**:

#### Form-Level Defaults
- Each form has default font settings that apply to all fields unless overridden:
  - **Font Family:** e.g., `Helvetica`, `Courier`, `Times-Roman` (default: `Helvetica`)
  - **Font Size:** in points, e.g., `10`, `12`, `14` (default: `12`)
- Form-level font settings are editable on the form editor page

#### Field-Level Overrides
- Individual fields can override the form-level defaults:
  - A field with no font override inherits the form's defaults
  - A field with a font override uses its own settings
- Useful for fields that need different sizing (e.g., a small "State" box vs a larger "Address" field)

#### Data Model
```json
{
  "form_id": "form-001",
  "font_family": "Helvetica",
  "font_size": 12,
  "fields": [
    {
      "field_name": "Name",
      "x": 15.2,
      "y": 22.5,
      "width": 30.0,
      "height": 3.5,
      "font_family": null,
      "font_size": null
    },
    {
      "field_name": "State",
      "x": 60.0,
      "y": 45.0,
      "width": 5.0,
      "height": 3.0,
      "font_family": null,
      "font_size": 9
    }
  ]
}
```

In this example, "Name" inherits `Helvetica 12pt` from the form. "State" inherits `Helvetica` but overrides font size to `9pt`.

### 5.7 Field Placement Versioning

- Every time a user saves field placements on a form, a **new version** is created
- The initial AI-generated placement is stored as version 1
- Each version is immutable once saved
- The form editor displays:
  - Current version number
  - A version history dropdown/list showing all versions with timestamps
  - A "Revert to this version" action for any previous version
- Reverting creates a **new version** (copy of the old one) rather than deleting intermediate versions
- Version data is stored in DynamoDB with a composite key: `formID` + `version`

#### Version Data Model
```json
{
  "form_id": "form-001",
  "version": 3,
  "created_at": "2026-03-03T14:30:00Z",
  "source": "manual_edit",
  "font_family": "Helvetica",
  "font_size": 12,
  "fields": [ ... ]
}
```

`source` values: `ai_analysis`, `manual_edit`, `revert_from_v1`

### 5.9 Form Population

- User selects a form from the library and a customer from the customer list
- The system renders the form with the customer's data placed at the saved field coordinates
- Text is rendered using the form-level font settings, with any field-level overrides applied
- The populated form can be previewed on-screen before downloading
- The user can download the populated form as a PDF
- Population uses the **latest version** of field placements stored for that form

### 5.10 Form Library

- Lists all uploaded forms with:
  - Thumbnail preview
  - Form name (editable)
  - Upload date
  - Field detection status (pending, complete, needs review)
  - Number of fields mapped
  - Current field placement version number
- Users can delete forms from the library
- Search and filter capabilities

## 6. Technical Architecture

### 6.1 Backend

- **Language:** Go
- **Framework:** GoFiber with the [GoFiber AWS Lambda adapter](https://github.com/gofiber/adaptor) for running in AWS Lambda
- **AI Service:** Amazon Bedrock — Claude Sonnet model for form analysis (already enabled in account)
- **File Storage:** AWS S3 for uploaded form files (pre-signed URL upload)
- **Data Storage:** Amazon DynamoDB for all application data

### 6.2 DynamoDB Tables

| Table | Partition Key | Sort Key | Description |
|-------|--------------|----------|-------------|
| `Users` | `email` (S) | — | User accounts with bcrypt password hashes |
| `Forms` | `id` (S) | — | Form metadata (name, status, upload date, S3 key, font settings) |
| `FieldPlacements` | `formId` (S) | `version` (N) | Versioned field placements per form |
| `Customers` | `id` (S) | — | Customer records |
| `Settings` | `key` (S) | — | App settings (standard field definitions, etc.) |
| `Sessions` | `token` (S) | — | Active session tokens with TTL |

### 6.3 Frontend

- Server-side rendered HTML templates (Go templates) served by GoFiber
- Vanilla JavaScript for interactive form editor (canvas-based field placement)
- CSS framework: Tailwind CSS (via CDN) for styling
- No heavy frontend framework to keep Lambda cold-start times low

### 6.4 Infrastructure & Deployment

- **Runtime:** AWS Lambda (single function, GoFiber handles routing)
- **API Gateway:** AWS API Gateway v2 (HTTP API) fronting the Lambda
- **File Storage:** AWS S3 bucket for form uploads (pre-signed URL access)
- **Database:** DynamoDB tables (defined in SAM template)
- **IaC:** AWS SAM (Serverless Application Model) template
- **CI/CD:** GitHub Actions workflow that:
  1. Runs tests
  2. Builds the Go binary for `linux/amd64`
  3. Packages with `sam build`
  4. Deploys with `sam deploy`
- **Region:** `us-east-1` (configurable)

## 7. AI Integration Details

### 7.1 Bedrock Request Flow

1. User uploads a form (PDF/image) via pre-signed S3 URL
2. User triggers analysis (or it auto-triggers on upload complete)
3. Backend fetches the form from S3
4. A request is sent to Bedrock with:
   - The form image (converted to base64 if needed)
   - A prompt instructing the model to identify field locations
   - The list of standard fields to detect
5. The model returns a JSON response containing:
   - Field name
   - Bounding box (x, y, width, height) as percentages of form dimensions
   - Confidence score (0.0 - 1.0)
   - Reasoning for the placement
6. Results are stored in DynamoDB (`FieldPlacements` table) as version 1 with source `ai_analysis`

### 7.2 Prompt Strategy

The prompt will instruct the AI to:
- Examine the form layout and identify labeled input areas
- Match each standard field to the most appropriate input area on the form
- Return coordinates as percentage-based positions relative to the form dimensions
- Provide confidence scores to flag uncertain placements
- If a field has no clear match on the form, indicate it with a low confidence score

### 7.3 Multi-Page Form Handling

- For multi-page PDFs, each page is analyzed independently
- Field placements include a `page` attribute
- The editor allows navigation between pages

## 9. Non-Functional Requirements

- **Cold Start:** Lambda cold start should remain under 3 seconds (Go binary advantage)
- **File Size:** Support form uploads up to 10MB (via pre-signed S3 URL)
- **Concurrency:** Handle up to 50 concurrent users (Lambda auto-scaling)
- **Browser Support:** Modern browsers (Chrome, Firefox, Safari, Edge — latest 2 versions)
- **Security:** Passwords bcrypt-hashed; sessions use secure HTTP-only cookies; S3 pre-signed URLs expire after 5 minutes

## 10. Out of Scope (Phase 1)

- Self-service user registration (admin seeds users directly)
- Role-based access control / permissions
- Multi-tenant data isolation
- Batch form population (fill multiple customers at once)
- Email or sharing of populated forms
- Mobile-optimized UI
- OCR for handwritten forms

## 11. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| AI field detection accuracy may be low for complex forms | Users spend time manually correcting | POC scope — manual editor is the primary safety net; confidence scores guide users |
| Lambda cold starts affect user experience | Slow initial page loads | Go binary minimizes cold start; consider provisioned concurrency for production |
| Large PDF processing may exceed Lambda timeout | Form analysis fails | Process asynchronously; increase Lambda timeout to 5 minutes for analysis |
| DynamoDB costs could grow with heavy versioning | Higher storage costs | TTL on old versions if needed; acceptable for POC scale |
| Single seed user with known password | Security risk if URL leaks | Change password after initial setup; acceptable for POC |

## 12. Success Metrics

- AI correctly places 70%+ of fields on common form types (W-9, W-4, insurance forms)
- Form population reduces manual entry time by 80%+ compared to filling forms by hand
- Users can upload a form and have it ready for population within 2 minutes
- Form editor is intuitive enough to adjust field placements in under 1 minute
- Version history allows users to confidently experiment with field placement changes
